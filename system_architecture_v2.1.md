# AI Secretary System Architecture v2 (Consolidated)

本仕様は、システムの「レスポンス高速化・無駄の削減」と「ランク（E〜S+）による高精度な状態管理」を両立するための最終設計図です。

## 📋 1. 基本コンセプト
- **「ローカル完結型」**: 常時処理は軽量な画面監視と State 更新のみ。**Gemini はユーザー入力時のみ稼働**。
- **「データ駆動型」**: すべての情報を `State` に集約し、その変化（イベント）をトリガーに独り言を生成。
- **「ターゲット OCR」**: 全画面 OCR を廃止。シーンに応じた特定座標の切り出し読み出しに特化。

## 🧠 2. ランク・数値・State設計

### 統合 State 構造 (データ一元管理)
- **Basic Info**: 馬名, 現在シーン
- **Stats (Rank & Value)**: スピード, スタミナ, パワー, 根性, 賢さ
- **Aptitude**: 芝/ダート (◎○△), 距離適性 (min-max), 脚質, 成長型
- **Condition**: 疲労, 調子, 絶好調〜不調フラグ
- **History**: 特性リスト, 直近レース結果, イベント履歴

## ⚙️ 3. 処理フロー

### ① スクリーンショット & 簡易判定
- 30秒間隔またはトリガーで取得。
- 前回との差分をハッシュで確認。変化があれば、ごく限定的な特徴量でシーンを判定。

### ② 限定領域 OCR & 正規化
- シーンに応じた矩形領域のみを切り出し OCR 実行。
- **正規化ロジック**: `Ｓ`→`S`, `＋`→`+` などの補正。
- **バリデーション**: ランクが不正な場合は State を更新せずフォールバック。

### ③ State 更新 & イベント検知
- 新旧 State を比較し、`stat_changed`, `aptitude_match`, `race_finished` 等のイベントを発火。

### ④ 独り言生成 (ローカル)
- 検知したイベントと State の数値に基づき、テンプレートやランダムロジックで発話。
- **例**: `if stamina.value >= 95: "スタミナSあるし、長距離も安心だね！♪"`

### ⑤ ユーザー入力時のみ Gemini 呼び出し
- 入力に応じて最適な情報を State から抽出し、専用テンプレートに流し込んで送信。

## 📑 4. Gemini 送信テンプレート例

### 【ローテーション提案 / Rotation】
> 以下の馬のローテーションを現実的に提案してください。
> 能力:{speed_rank}/{stamina_rank}等, 適性:{min}〜{max}, 脚質:{style}, 疲労:{fatigue}...

### 【評価・予想 / Evaluation & Prediction】
> この馬の特徴と強みを評価 / レース適性を予想してください。
> (State から抽出した能力・適性・特性データを添付)

## 🚀 5. 設計上の重要件
- **ランク保存**: Gemini には理解しやすい「ランク」を渡し、内部判断には「数値」を使う。
- **負荷低減**: 通常ループでは GeminiBrowser はアイドル状態を維持。
- **精度重視**: 「+」の誤認識補正を徹底し、データの整合性を守る。
